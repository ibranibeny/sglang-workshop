Deploy SGLang on Azure H100

Build a secure, OpenAI-compatible endpoint for Qwen3.6-35B-A3B-FP8, served by SGLang on one Azure H100 NVL GPU and protected by Caddy HTTPS plus API-key authentication.

Document type: Tutorial (Diátaxis)
Audience: DevOps and ML engineers
Duration: Approximately 2 hours
Level: L200
Final API: https://<your-domain>/v1
Updated: 18 July 2026

1. Objectives

After completing this workshop, you can:

Provision

Create an Azure Standard_NC40ads_H100_v5 VM in indonesiacentral.

Serve

Run Qwen/Qwen3.6-35B-A3B-FP8 with SGLang and a 262144-token context configuration.

Secure

Keep SGLang on loopback and expose only the Caddy HTTPS gateway with secret-key enforcement.

Verify

Prove health, model discovery, chat completion, and structured tool calling end to end.

Tutorial rule: follow the workshop in order. Every deployment step ends with a checkpoint that must pass before you continue.

Understand the system

3. Concept overview

The deployment combines four components with one responsibility each:

Azure H100 VM

One NVIDIA H100 NVL with 94 GB VRAM runs the model. The exact default SKU is Standard_NC40ads_H100_v5.

SGLang

Provides the OpenAI-compatible API, reasoning parser qwen3, and tool parser qwen3_coder.

Caddy

Terminates HTTPS, renews Let's Encrypt certificates, checks the secret key, and proxies accepted requests.

Network boundary

SGLang publishes port 30000 only on 127.0.0.1. Clients reach Caddy on HTTPS instead of the model server directly.

Authentication contract

The gateway accepts either Authorization: Bearer <key> for OpenAI-compatible clients or api-key: <key> for Foundry ModelGateway. The unauthenticated /health path remains available for uptime checks.

Follow the request

4. Architecture

The client resolves your DNS name, reaches the VM through the required ports, authenticates at Caddy, and only then reaches SGLang on loopback.

flowchart LR Client[Client or OpenAI SDK] -->|HTTPS 443| DNS[DNS name] DNS --> NSG[Azure NSG] subgraph VM[Azure NC40ads H100 v5 VM] Caddy[Caddy: TLS and API key] SGLang[SGLang: 127.0.0.1:30000] GPU[NVIDIA H100 NVL 94 GB] Cache[Hugging Face cache] Caddy -->|Authenticated proxy| SGLang SGLang --> GPU SGLang --> Cache end NSG --> Caddy

Port 80 is required for ACME HTTP challenges. Port 22 is optional and environment-dependent; this workshop uses Azure Run Command when corporate policy blocks SSH.

Authenticated request flow

sequenceDiagram participant Client participant Caddy participant SGLang participant H100 Client->>Caddy: POST /v1/chat/completions plus secret key alt Key missing or invalid Caddy-->>Client: 401 Unauthorized else Key accepted Caddy->>SGLang: Proxy to loopback:30000 SGLang->>H100: Run inference H100-->>SGLang: Generated tokens or tool call SGLang-->>Caddy: OpenAI-compatible JSON Caddy-->>Client: 200 OK end

5. Guided deployment

Verify prerequisites and quota

Prove the operator environment is ready before Azure incurs cost.

Purpose

Confirm required local tools, Azure authentication, the selected subscription, and H100-family quota.

Run

git --version
openssl version
az version
az account show --query "{subscription:name, tenant:tenantId}" -o table
az vm list-usage -l indonesiacentral -o table

Expected result

Each tool reports a version, Azure displays the intended subscription, and the regional usage table includes enough available NCads H100 v5 family vCPUs for the 40-vCPU VM.

Checkpoint

Do not continue until authentication works and quota for Standard_NC40ads_H100_v5 is confirmed in indonesiacentral.

Clone the deployment repository

Start with the tested scripts rather than copying isolated commands.

Purpose

Obtain the central configuration, lifecycle scripts, Caddy template, and endpoint validation suite.

Run

git clone https://github.com/ibranibeny/sglang-azure-workshop.git
cd sglang-azure-workshop/deploy

Expected result

The repository clones successfully and the four listed files are present in deploy/.

Checkpoint

Your current directory is sglang-azure-workshop/deploy, and config.sh, 00-genkey.sh, and 05-run-sglang.sh exist.

2

Configure the custom domain and deployment values

Set the desired endpoint identity before provisioning.

Purpose

Review the single source of configuration and override only values that differ in your environment.

Run

# Review defaults
source ./config.sh
printf 'Region: %s\nVM: %s\nModel: %s\nDomain: %s\n' \
  "$LOCATION" "$VM_SIZE" "$MODEL_PATH" "$TLS_DOMAIN"

# Example overrides for this shell session
export TLS_DOMAIN="openai.example.com"
export TLS_EMAIL="operator@example.com"
# Export only when model access requires it:
# export HF_TOKEN="hf_..."

Expected result

The defaults show indonesiacentral, Standard_NC40ads_H100_v5, and Qwen/Qwen3.6-35B-A3B-FP8. The repository's configured example domain is openai.contoso.day; replace it with a DNS name you control.

Checkpoint

TLS_DOMAIN contains the intended custom domain. After Step 4 returns the static IP, create a DNS-only A record from this name to that IP before launching Caddy.

3

Generate the API key

Create a 256-bit secret without printing it to terminal output.

Purpose

Protect every OpenAI API path before the endpoint becomes reachable.

Run

bash 00-genkey.sh
ls -l .secrets/api_key

git check-ignore .secrets/api_key
# Later rotation: bash 00-genkey.sh --rotate

Expected result

The script reports the secret file location but not its value. The file is stored under the git-ignored deploy/.secrets/ directory and uses mode 600 where supported.

Checkpoint

.secrets/api_key exists and git check-ignore confirms it cannot be committed. Keep the key private.

4

Provision the Azure H100 VM

Create the resource group, network, static IP, VM, and GPU driver.

Purpose

Provision the Azure foundation idempotently from the repository's central configuration.

Run

bash 01-deploy.sh

az vm extension show \
  -g sglang-rg --vm-name sglang-h100 \
  -n NvidiaGpuDriverLinux \
  --query provisioningState -o tsv

Expected result

The script creates or reuses the resource group, VNet, NSG, static public IP, NIC, Ubuntu VM, and Microsoft.HpcCompute/NvidiaGpuDriverLinux extension. It prints the assigned IP locally; do not paste that value into documentation.

Checkpoint

The extension reports Succeeded. Create the DNS-only A record described in Step 2, then immediately narrow the initial broad testing rule in Step 6.

5

Verify or start the VM

Separate compute power state from application readiness.

Purpose

Ensure Azure allocated the VM before container startup and GPU validation.

Run

bash 02-start-vm.sh

az vm get-instance-view -g sglang-rg -n sglang-h100 \
  --query "instanceView.statuses[?starts_with(code, 'PowerState/')].code" \
  -o tsv

Expected result

The start script is safe to rerun and the instance-view query returns PowerState/running.

Checkpoint

The VM reports PowerState/running. This confirms compute state only; Step 7 confirms model readiness.

6

Apply the required network rules

Replace the broad bootstrap rule with explicit service ports.

Purpose

Expose Caddy HTTPS and certificate validation without exposing SGLang's port 30000.

Run

bash 03b-open-nsg-ports.sh

az network nsg rule list -g sglang-rg --nsg-name sglang-nsg \
  --query "[?direction=='Inbound'].{name:name, port:destinationPortRange, access:access, priority:priority}" \
  -o table

Expected result

Narrow inbound rules exist for HTTPS 443 and ACME 80. SSH 22 is optional for administration, but corporate GSA may block it. The legacy 03-open-nsg.sh is broad and testing-only; 04-check-nsg.sh checks that old allow-all design, not these narrow rules.

Checkpoint

The required ports 443 and 80 are allowed, port 30000 is not public, and DNS resolves to the VM's static IP. Use 03b-open-nsg-ports.sh again if automated hardening removes the rules.

7

Launch SGLang and Caddy

Download the model, start inference, and wait for loopback health.

Purpose

Run the model on the H100 and expose only the authenticated HTTPS gateway.

Run

bash 05-run-sglang.sh

# Default readiness budget used by the script:
READINESS_ATTEMPTS=60
READINESS_INTERVAL=30

# If readiness needs investigation:
az vm run-command invoke -g sglang-rg -n sglang-h100 \
  --command-id RunShellScript \
  --scripts "docker ps; docker logs --tail 80 sglang; nvidia-smi" \
  --query "value[0].message" -o tsv

Expected result

The script launches SGLang with --context-length 262144, reasoning parser qwen3, tool parser qwen3_coder, and Caddy on HTTPS. Readiness is checked on the VM at http://127.0.0.1:30000/health, avoiding GSA and hairpin-NAT false failures.

Checkpoint

The output includes SGLANG_READY. If not, inspect model logs and GPU state through Azure Run Command before continuing.

8

Validate the endpoint end to end

Test from Windows PowerShell when the corporate network blocks WSL.

Purpose

Prove that public HTTPS, authentication, OpenAI compatibility, generation, and tool calling work together.

Run — Windows PowerShell from the cloned repository root

cd deploy
.\test-tool-calling.ps1

Expected result

The suite reports PASS for health, models, and basic chat, then prints a get_weather tool call and TOOL_CALLING_WORKS. Caddy accepts both Authorization: Bearer <key> and the Foundry-compatible api-key header.

Checkpoint

All four test stages pass. A WSL public-endpoint failure alone is not evidence of an Azure failure when Global Secure Access is active.

Collect evidence

6. Validation evidence

The repository's PowerShell suite validates the endpoint in four layers:

1. Health

GET /health returns HTTP 200 without authentication.

2. Models

GET /v1/models returns the served model with the secret header.

3. Chat

POST /v1/chat/completions returns a valid completion.

4. Tool calling

The model returns structured tool_calls and TOOL_CALLING_WORKS.

PASS health HTTP 200
PASS model id: Qwen/Qwen3.6-35B-A3B-FP8
PASS chat content: ...
tool: get_weather  args: {"city":"Jakarta"}
PASS TOOL_CALLING_WORKS

Recover systematically

7. Troubleshooting

WSL cannot reach the domain

Cause: corporate Global Secure Access blocks WSL outbound traffic. Recovery: test the public endpoint from Windows PowerShell; use Azure Run Command for VM-local checks.

HTTPS connection closes

Cause: the narrow NSG rules may have been removed. Recovery: run 03b-open-nsg-ports.sh, verify 80/443, and test again.

TLS handshake fails after NSG recovery

Cause: Caddy entered ACME backoff while ports were closed. Recovery: restore rules, then restart Caddy through az vm run-command so certificate issuance retries immediately.

Model is not ready

Cause: first-run weight download or GPU initialization. Recovery: inspect docker logs --tail 80 sglang and nvidia-smi through Azure Run Command.

API returns 401

Cause: missing or stale key. Recovery: load .secrets/api_key; after rotation, rerun 05-run-sglang.sh to restart Caddy without reloading the model.

Tool call is plain text

Cause: parser mismatch or stale container. Recovery: verify --tool-call-parser qwen3_coder, then force a model-container restart if configuration changed.

Workshop gate

8. Completion checklist

  • The Azure VM uses Standard_NC40ads_H100_v5 in indonesiacentral.
  • The NVIDIA driver reports a healthy H100 NVL GPU.
  • SGLang health returns HTTP 200 on VM loopback port 30000.
  • Caddy serves the custom domain over HTTPS.
  • Port 30000 is not exposed publicly; only required service ports are open.
  • Unauthenticated OpenAI API requests return 401.
  • Authenticated model discovery and chat completion pass.
  • The tool-calling test returns TOOL_CALLING_WORKS.
  • The API key remains in the git-ignored secrets directory.
  • The VM is deliberately retained, deallocated, or deleted after the lab.
