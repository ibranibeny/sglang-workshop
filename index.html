1. Overview & Objectives

This workshop demonstrates how to deploy a large language model (LLM) inference endpoint on Azure infrastructure using SGLang as the serving engine and Caddy as a reverse proxy with HTTPS termination. The result is a fully functional OpenAI-compatible API endpoint accessible over public HTTPS.

Learning Objectives

  • Provision an Azure VM with NVIDIA H100 GPU (Standard_NC40ads_H100_v5)
  • Deploy SGLang to serve the Qwen3.6-35B-A3B-FP8 model with FP8 quantization
  • Configure Caddy as an HTTPS reverse proxy with Let's Encrypt TLS certificates
  • Enforce API-key authentication on all inference requests
  • Configure Azure Network Security Group (NSG) rules for controlled access
  • Verify end-to-end inference including tool-calling capabilities
  • Understand the networking considerations: Global Secure Access (GSA) and WSL outbound restrictions

What You'll Build

A public HTTPS endpoint at https://openai.contoso.day/v1 that accepts OpenAI-compatible chat completion requests, enforces API-key authentication, and routes inference to a Qwen3.6 model running on an H100 GPU.

https://openai.contoso.day/v1/chat/completions

4. Deployment Guidance

The deployment is fully automated through a series of bash scripts in the deploy/ directory. Each script handles a specific phase of the infrastructure and application lifecycle.

Phase 1: Setup

1

Generate API Key

deploy/00-genkey.sh — Generates a 64-character random API key and stores it in deploy/.secrets/api_key (git-ignored, mode 600).

cd deploy
./00-genkey.sh
# Output: API_KEY=xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
# Stored in: .secrets/api_key
2

Provision VM + GPU Driver

deploy/01-deploy.sh — Creates the resource group, VNet, and H100 VM. Installs NVIDIA GPU driver via Azure Extension. Configures cloud-init for Docker installation.

./01-deploy.sh
# Creates:
#   - Resource Group: sglang-rg
#   - VM: sglang-h100 (Standard_NC40ads_H100_v5)
#   - VNet: sglang-vnet
#   - Public IP: [assigned] (static)
#   - NVIDIA GPU Driver extension

Phase 2: Network

3

Ensure VM is Running

deploy/02-start-vm.sh — Verifies the VM is in a running state before proceeding.

./02-start-vm.sh
# Waits until provisioningState=Succeeded
4

Open NSG Firewall

deploy/03-open-nsg.sh — Creates NSG inbound rules for SSH (22), HTTP/ACME (80), and HTTPS (443). Important: If Azure Defender strips these rules, use 03b-open-nsg-ports.sh to restore them.

./03-open-nsg.sh
# Creates NSG rules:
#   - AllowSSH  (port 22)
#   - AllowACME (port 80)  — for Let's Encrypt challenges
#   - AllowHTTPS (port 443)

# If rules are stripped by Azure Defender:
./03b-open-nsg-ports.sh

Phase 3: Application

5

Launch SGLang + Caddy

deploy/05-run-sglang.sh — Pulls and starts both Docker containers. Performs a readiness health check on the VM's loopback interface (bypassing WSL/GSA network restrictions).

./05-run-sglang.sh
# Launches:
#   - sglang-server: Qwen/Qwen3.6-35B-A3B-FP8 on 127.0.0.1:30000
#   - caddy: HTTPS reverse proxy on :443
#
# Health check: loops curl http://127.0.0.1:30000/health
# via az vm run-command (runs on VM, bypasses GSA)
#
# Tunable:
#   READINESS_ATTEMPTS=900   (default 30 min)
#   READINESS_INTERVAL=2     (seconds)

Cleanup

✕

Delete Everything

deploy/99-destroy.sh — Deletes the resource group and all associated resources.

./99-destroy.sh
# WARNING: This permanently deletes:
#   - Resource Group: sglang-rg
#   - VM, NSG, VNet, Public IP, all disks
#   - All data is lost

5. Testing the Endpoint

From Windows PowerShell (not WSL — GSA blocks WSL outbound to public domains):

# Load API key
$API_KEY = Get-Content deploy\.secrets\api_key

# Health check (no auth required)
curl https://openai.contoso.day/health

# List models (authenticated)
curl https://openai.contoso.day/v1/models `
  -H "Authorization: Bearer $API_KEY"

# Chat completion (authenticated)
curl https://openai.contoso.day/v1/chat/completions `
  -H "Authorization: Bearer $API_KEY" `
  -H 'Content-Type: application/json' `
  -d '{"model":"Qwen/Qwen3.6-35B-A3B-FP8","messages":[{"role":"user","content":"Hello"}]}'

# Run full test suite
.\deploy\test-tool-calling.ps1
⚠️ TLS 1.2 Required: PowerShell 5.1 defaults to TLS 1.0. Always add [Net.ServicePointManager]::SecurityProtocol = [Net.SecurityProtocolType]::Tls12 before making HTTPS requests.

6. Troubleshooting

Endpoint unreachable from WSL

Azure Global Secure Access (GSA) blocks WSL outbound traffic to public domains. Use Windows PowerShell or SSH into the VM to test.

TLS handshake failure ("Could not create SSL/TLS secure channel")

Caddy's Let's Encrypt ACME challenge may have failed while NSG was down. Restart Caddy to retry:

az vm run-command invoke -g sglang-rg -n sglang-h100 \
  --command-id RunShellScript \
  --scripts "docker restart caddy"

NSG rules missing (Azure Defender adaptive hardening)

Azure Defender can strip NSG inbound rules. Restore them:

./deploy/03b-open-nsg-ports.sh

Check VM health directly

az vm run-command invoke -g sglang-rg -n sglang-h100 \
  --command-id RunShellScript \
  --scripts "docker ps && curl -s http://127.0.0.1:30000/health"

8. Custom Domain Configuration

If you want to use your own domain instead of openai.contoso.day, follow these steps to configure DNS and update the deployment.

Step 1: Configure DNS Record

Add an A record in your DNS provider's control panel pointing to your Azure VM's public IP:

Record TypeNameValueTTL
Aopenai (or @ for apex)[Your Azure Public IP]Auto
💡 Cloudflare Users: Set proxy status to DNS-only (grey cloud) to avoid routing issues. Do not enable CDN or WAF — Caddy handles TLS directly.

Step 2: Update Deployment Configuration

Edit deploy/config.sh to set your custom domain:

# Edit deploy/config.sh
DOMAIN="yourdomain.com"          # Your custom domain
SUBDOMAIN="openai"               # Subdomain (optional)
FULL_DOMAIN="${SUBDOMAIN}.${DOMAIN}"  # e.g., openai.yourdomain.com

Step 3: Update Caddy Configuration

Caddy automatically requests Let's Encrypt certificates for your domain. Ensure the Caddyfile.tmpl references the correct domain:

# In deploy/Caddyfile.tmpl
{
    ${FULL_DOMAIN} {
        reverse_proxy 127.0.0.1:30000
        
        header {
            Strict-Transport-Security "max-age=31536000; includeSubDomains"
            X-Content-Type-Options "nosniff"
        }
        
        # API Key authentication
        @auth header Authorization /^Bearer \\s+YOUR_API_KEY\\s*$/
        respond @auth "Unauthorized" 401
    }
}

Step 4: Redeploy

After updating the configuration, restart Caddy to apply the new domain and request a fresh TLS certificate:

# SSH into the VM and restart Caddy
az vm run-command invoke -g sglang-rg -n sglang-h100 \
  --command-id RunShellScript \
  --scripts "docker restart caddy"

# Verify TLS certificate
curl -I https://yourdomain.com/v1/health

Verification

Confirm your custom domain is working correctly:

CheckCommandExpected Result
DNS Resolutionnslookup yourdomain.comResolves to Azure Public IP
TLS Certificateopenssl s_client -connect yourdomain.com:443 -servername yourdomain.comCertificate issued for yourdomain.com
Health Endpointcurl https://yourdomain.com/healthHTTP 200 OK

⚠️ Important: Let's Encrypt certificates are valid for 90 days. Caddy automatically renews them, but ensure port 80 remains open for ACME challenges. If using a reverse proxy or CDN, configure it to pass ACME challenges through to Caddy.
